# 実装計画：リアルタイム議事録共同編集システム MVP

## 1. この文書の目的

この文書は `docs/system-requirements.md` をもとに、実装をどの順番で進めるかを整理した作業計画です。

目的は、最小限で動く MVP を早く確認し、その後に不足を足していけるようにすることです。

## 2. 実装方針

最初から完成形を目指さず、次の順で段階的に作る。

1. Chrome 拡張機能の最小 UI を作る
2. ローカル中継アプリを起動できるようにする
3. Chrome 拡張機能からローカル中継アプリへ接続する
4. Google Meet タブ音声を送れるようにする
5. 音声認識 API とつなぐ
6. Google ドキュメントへ追記する
7. 失敗時の再試行や状態表示を整える

## 3. 実装フェーズ

### Phase 1: プロジェクト土台作成

目的:
最低限のファイル構成を作り、拡張機能とローカル中継アプリを別々に開発できる状態にする。

実施内容:

- Chrome 拡張機能用ディレクトリを作る
- ローカル中継アプリ用ディレクトリを作る
- 各ディレクトリの責務を README かコメントで明記する
- 実行方法を簡単に確認できるようにする

完了条件:

- 拡張機能側とローカル中継側の配置が決まっている
- 今後どこを編集すればよいか迷わない

### Phase 2: Chrome 拡張機能の最小 UI

目的:
ホストが Google ドキュメント URL を入力し、記録開始と停止を操作できるようにする。

実施内容:

- ポップアップ画面を作る
- Google ドキュメント URL 入力欄を作る
- 記録開始ボタンを作る
- 記録停止ボタンを作る
- 状態表示欄を作る

完了条件:

- 拡張機能を Chrome に読み込める
- ポップアップから基本操作ができる

### Phase 3: Google ドキュメント URL 確認

目的:
URL 入力方式で安定して対象ドキュメントを指定できるようにする。

実施内容:

- Google ドキュメント URL 形式を検証する
- URL からドキュメント ID を抽出する
- 不正な URL の場合はエラー表示する
- 正常な場合は一時保存できるようにする

完了条件:

- ユーザーが URL を貼り付けるだけで対象ドキュメントを指定できる
- 入力ミス時に分かりやすいエラーが出る

### Phase 4: ローカル中継アプリの最小起動

目的:
ホスト PC 上でローカル中継アプリを起動し、Chrome 拡張機能から接続できるようにする。

実施内容:

- Node.js でローカル中継アプリを作る
- `localhost WebSocket` で待ち受ける
- 接続、切断、エラーをログ出力する
- 最小限のヘルスチェック方法を決める

完了条件:

- ローカル中継アプリが起動する
- Chrome 拡張機能から接続確認できる

### Phase 5: 拡張機能とローカル中継アプリの接続

目的:
記録開始と停止に応じて、両者の通信を制御できるようにする。

実施内容:

- 記録開始時に WebSocket 接続する
- 記録停止時に切断する
- 接続成功、失敗、再接続中の状態を表示する
- 制御メッセージの形式を決める

完了条件:

- 記録開始、停止に応じて接続状態が変わる
- 状態表示が実際の状態と一致する

### Phase 6: Google Meet タブ音声の取得

目的:
Chrome 拡張機能で Google Meet タブ音声を取得し、ローカル中継アプリへ送れるようにする。

実施内容:

- 対象タブの識別方法を決める
- TabCapture API で音声を取得する
- 必要なら音声データをチャンク化する
- 音声チャンクをローカル中継アプリへ送る

完了条件:

- Google Meet タブ音声がローカル中継アプリへ届く
- 無音時や停止時の挙動が極端に不安定でない

### Phase 7: 音声認識 API 連携

目的:
受信した音声を文字起こしできるようにする。

実施内容:

- 音声認識 API への接続処理を作る
- 音声データを必要形式へ整形する
- 確定済み文字起こしだけを抽出する
- 一時失敗時に再試行できるようにする

完了条件:

- 会議音声から文字起こし結果が得られる
- 未確定文ではなく確定済みテキストを扱える

### Phase 8: Google ドキュメント追記

目的:
文字起こし結果を共同 transcript へ自動追記できるようにする。

実施内容:

- Google Docs API の認証を行う
- バッファリング処理を実装する
- 数秒ごとに末尾追記する
- 書き込み失敗時は再試行する
- 重複追記を防ぐ

完了条件:

- 確定済み文字起こしが Google ドキュメントへ追記される
- 既存行の再上書きが発生しない

### Phase 9: 人間修正を守る調整

目的:
共同編集のしやすさを壊さないように仕上げる。

実施内容:

- 追記が常に末尾になることを確認する
- 人間が修正した既存行を再上書きしないことを確認する
- 補足記号の扱いを確認する
- ドキュメント共有設定の案内文を整える

完了条件:

- 人間とシステムの編集がぶつかりにくい
- 会議中の運用手順が説明できる

### Phase 10: 動作確認

目的:
MVP として一連の流れが成立しているか確認する。

実施内容:

- Google ドキュメント URL 入力
- ローカル中継アプリ起動
- 記録開始
- Google Meet 音声取得
- 文字起こし
- Google ドキュメント追記
- 記録停止

完了条件:

- 最初から最後まで手動操作で通せる
- 主なエラー時の表示が確認できる

手順とエラーパターンの具体例は [mvp-operation-checklist.md](mvp-operation-checklist.md) にまとめた。

## 4. 優先順位

最優先:

- URL 指定
- 記録開始、停止
- ローカル中継接続
- 音声取得
- 文字起こし
- Google ドキュメント追記

後回しでよいもの:

- 見た目の改善
- 前回 URL の再利用
- 詳細ログ保存基盤
- Cloud Run 対応

## 5. 実装時の注意

- 人間が修正した既存行は再上書きしない
- 話者分離は MVP に入れない
- Meet 字幕 DOM 取得は使わない
- 出力先指定は URL 入力に固定する
- まずは 1 人で通し確認できる最小構成を優先する

## 6. MVP 完了の定義

次の条件を満たしたら MVP 完了とする。

- Google Meet 会議中に文字起こし結果が Google ドキュメントへ数秒単位で追記される
- 参加者が同じドキュメント内で誤字修正できる
- システムが既存行を勝手に上書きしない
