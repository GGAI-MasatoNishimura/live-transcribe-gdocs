# システム要件定義：リアルタイム議事録共同編集システム

## 1. 目的

本システムの目的は、Google Meet でのオンライン会議中に会話内容をリアルタイムで文字起こしし、Google ドキュメントへ自動追記することです。
同じドキュメントを会議参加者が共同編集し、会議終了時点で正確な会議ログが過不足なくそろっている状態を目指します。

## 2. MVP の前提

- 対象会議ツール：Google Meet（ブラウザ版）
- 実装形態：Google Chrome 拡張機能 + ホスト PC 上のローカル中継アプリ
- 出力先：Google ドキュメント
- リアルタイム処理対象：文字起こし
- 話者分離、話者名推定：対象外

## 3. 採用アーキテクチャ

### 3.1 全体構成

- Chrome 拡張機能
  - Google Meet タブの音声を取得する
  - 記録開始、記録停止の操作を提供する
  - 音声データをローカル中継アプリへ送る
- ローカル中継アプリ
  - Chrome 拡張機能から音声を受け取る
  - 音声認識 API へ送信する
  - 受信した文字起こし結果を一時バッファへためる
  - 数秒ごとに Google ドキュメントへ追記する
  - エラー時の再試行、ログ出力を担当する
- Google ドキュメント
  - 単一の共同 transcript エリアを中心に運用する
  - システムは確定済み文字起こしを末尾へ追記する
  - 人間は同じエリア内で誤字や聞き取り漏れを修正する

## 4. 使用技術

- 音声取得
  - Chrome TabCapture API
  - Web Audio API
- ローカル中継
  - Node.js を第一候補とする
  - Chrome 拡張機能との通信は localhost WebSocket を第一候補とする
- 音声認識
  - Google Cloud Speech-to-Text API を第一候補とする
- 文書操作
  - Google Docs API `documents.batchUpdate`
- 認証
  - Google OAuth 2.0

## 5. Google ドキュメントの構成

Google ドキュメントは、会議中に共同で育てる transcript を主役とする。

### 5.1 主エリア

ドキュメント本文を単一の共同 transcript エリアとして扱う。

- システムは確定済み文字起こしを末尾へ追記する
- 人間は必要に応じて誤字、脱字、聞き取り違いを修正する
- 人間は必要に応じて短い補足を書き加えてよい
- システムは一度追記した既存行を再上書きしない

### 5.2 先頭ヘッダ

ドキュメント先頭には最低限の会議メタ情報を置く。

想定する記載項目は次の通り。

- 会議名
- 開催日時
- 参加者
- 補足ルール

ここでいう補足ルールとは、人間が補足を書く際の簡単な書き方を指す。

例:

- `[補足]`
- `[要確認]`

## 6. Google ドキュメントの指定方法

MVP では、出力先の Google ドキュメントは Chrome 拡張機能から指定する。

### 6.1 指定方法

- ホストは出力先の Google ドキュメント URL を Chrome 拡張機能へ入力する
- Chrome 拡張機能は入力された URL からドキュメント ID を自動抽出する
- ドキュメント ID をユーザーが手入力する方式は採用しない

### 6.2 この方式を採用する理由

- ユーザーにとって URL の貼り付けが最も分かりやすいため
- ドキュメント ID の切り出しを人手に任せると入力ミスが起きやすいため
- 将来、前回入力値の再利用機能を追加しやすいため

### 6.3 記録開始前の確認

Chrome 拡張機能またはローカル中継アプリは、記録開始前に次の確認を行う。

- 入力値が Google ドキュメントの URL 形式であること
- URL からドキュメント ID を抽出できること
- ホストの認証状態で対象ドキュメントへ書き込み可能であること

### 6.4 共有設定に関する注意

Google ドキュメントを指定できても、匿名参加者が共同編集できるとは限らない。

そのため、ホストは次の条件を満たす必要がある。

- 対象ドキュメントの共有設定がリンク編集可能であること
- 利用組織のポリシー上、匿名または未ログイン参加者の編集が許可されていること

### 6.5 将来の拡張候補

MVP では URL 入力方式を採用するが、将来は次の方式を追加候補とする。

- 前回使ったドキュメント URL の再利用
- Google Drive 一覧からの選択

## 7. 処理フロー

### 7.1 記録開始

1. ホストが Chrome 拡張機能へ Google ドキュメント URL を入力する
2. システムが URL 形式、ドキュメント ID、書き込み可否を確認する
3. ホストが Chrome 拡張機能から記録開始を実行する
4. Chrome 拡張機能が対象の Google Meet タブ音声を取得する
5. 取得した音声をローカル中継アプリへ送信する

### 7.2 リアルタイム文字起こし

1. ローカル中継アプリが音声データを受信する
2. 音声認識 API へ継続的に送信する
3. API から返ってきた文字起こし結果のうち、確定済みテキストのみを採用する

ここでいう確定済みテキストとは、あとから内容が大きく変わらない確定結果を指す。

### 7.3 バッファリング

Google Docs API の呼び出し回数を抑えるため、文字起こし結果は一時バッファへためる。

- 追記は都度ではなく数秒単位でまとめて行う
- 追記間隔は 3 秒前後を初期値とする

### 7.4 ドキュメント更新

1. バッファに未反映テキストがあるか確認する
2. 未反映テキストがある場合のみ Google Docs API を呼び出す
3. 共同 transcript エリアの末尾へ追記する
4. 書き込み成功後に対象バッファをクリアする

## 8. 人間によるリアルタイム修正

本 MVP では、AI や音声認識結果に誤字があった場合に、人間がその場で修正できることを重視する。

目的は、会議終了時点で正確な会議ログを完成させることである。

そのための仕様は次の通り。

- システムは未確定文をドキュメントへ書かない
- システムは確定済みテキストだけを追記する
- システムは一度確定して書き込んだ既存行を再編集しない
- 人間は共同 transcript エリアの既存行を自由に修正してよい
- 人間は必要に応じて短い補足や要確認メモを追加してよい

## 9. 対象外とする項目

MVP では次の項目を実装対象外とする。

- 話者分離
- 実名レベルの話者識別
- Google Meet の字幕 DOM 取得
- Meet Media API を使った参加者単位メディア取得

## 10. 匿名共同編集に関する前提

参加者が Google アカウントを持っていなくても共同編集できるようにするため、Google ドキュメントの共有設定はリンク編集可能を前提とする。

ただし、これは利用環境によって制限される可能性がある。

- Google ドライブ共有設定で「リンクを知っている全員が編集可能」が許可されていること
- 組織ポリシーで匿名編集が禁止されていないこと
- 同時編集人数が Google ドキュメントの上限を超えないこと

## 11. 非機能要件

### 11.1 安定性

- ローカル中継アプリが一時的に失敗しても、会議全体が止まらないこと
- 音声認識 API エラー時に再試行できること
- Google Docs API 書き込み失敗時に再試行できること

### 11.2 運用性

- ホストは会議開始前に記録開始できること
- 記録中の状態がホストに分かること
- 記録停止を明示的に行えること

## 12. コンポーネント責務

MVP では、各コンポーネントの責務を明確に分ける。

### 12.1 Chrome 拡張機能の責務

Chrome 拡張機能は、会議音声の取得とローカル中継アプリへの送信に責務を限定する。

担当する内容は次の通り。

- 記録開始、記録停止の UI を提供する
- 対象の Google Meet タブを識別する
- Google Meet タブ音声を取得する
- 音声データをローカル中継アプリへ送信する
- 接続状態、記録状態、エラー状態をホストへ表示する

担当しない内容は次の通り。

- Google Docs API への直接書き込み
- 音声認識 API の再試行制御
- 長期ログ保存

### 12.2 ローカル中継アプリの責務

ローカル中継アプリは、リアルタイム処理の中核を担う。

担当する内容は次の通り。

- Chrome 拡張機能から音声データを受信する
- 音声データを音声認識 API に適した形式へ変換または整形する
- 音声認識 API との接続を管理する
- 確定済み文字起こしを抽出する
- 文字起こし結果をバッファへためる
- バッファ内容を Google Docs API で追記する
- 再試行、エラーログ、状態ログを管理する

### 12.3 Google ドキュメントの責務

Google ドキュメントは、会議中に共同で育てる transcript の保存先として使う。

担当する内容は次の通り。

- リアルタイム文字起こしの共同編集先になる
- 人間が誤字や聞き取り漏れを補正する場所になる

## 13. 拡張機能とローカル中継アプリの通信

Chrome 拡張機能とローカル中継アプリの間には、将来 Cloud Run へ移行しやすいように、役割のはっきりした通信境界を置く。

### 13.1 接続方式

- MVP では `localhost WebSocket` を第一候補とする
- ローカル中継アプリはホスト PC 上で待ち受ける
- Chrome 拡張機能は記録開始時にローカル中継アプリへ接続する

### 13.2 送信する主なデータ

Chrome 拡張機能からローカル中継アプリへ送る主なデータは次の通り。

- セッション開始通知
- セッション終了通知
- 音声チャンク
- 対象ドキュメント識別子
- 記録状態に関するメタ情報

ここでいう音声チャンクとは、連続音声を小さな単位に区切った送信データを指す。

### 13.3 返却する主なデータ

ローカル中継アプリから Chrome 拡張機能へ返す主なデータは次の通り。

- 接続成功
- 記録中
- 再接続中
- エラー発生
- 停止完了

### 13.4 重要な設計方針

- 音声データと制御メッセージを区別する
- Chrome 拡張機能は音声処理ロジックを持ちすぎない
- ローカル中継アプリは UI に依存しない
- 将来 Cloud Run へ移すときも、できるだけ同じメッセージ形式を使う

## 14. 文字起こし追記ルール

人間による修正を壊さないため、追記ルールを固定する。

### 14.1 追記単位

- 追記対象は確定済み文字起こしのみ
- 追記先はドキュメント本文の末尾のみ
- 既存テキストの置換や途中挿入は行わない

### 14.2 表記の考え方

MVP では、話者分離を行わない前提で単純な行追加を優先する。

基本の考え方は次の通り。

- 会話順を崩さない
- 人間が目で追いやすい単位で改行する
- 過度な装飾は入れない

### 14.3 人間の修正を守るための制約

- システムは既存行の再整形をしない
- システムは人間が追記した補足記号を消さない
- システムは誤字修正済みの箇所を再上書きしない

## 15. 例外時の扱い

MVP でも最低限の異常系を定義しておく。

### 15.1 ローカル中継アプリ未起動

- Chrome 拡張機能は接続失敗を表示する
- ホストはローカル中継アプリの起動を求められる
- 音声取得だけ開始して先に進むことはしない

### 15.2 音声認識 API 一時失敗

- ローカル中継アプリが再試行する
- ホストには劣化中または再接続中であることを表示する
- 再試行中の失敗内容はログへ残す

### 15.3 Google Docs API 書き込み失敗

- 未反映テキストはバッファに保持する
- 書き込み成功まで再試行する
- 重複追記を防ぐようにする

## 16. Cloud Run への移行前提

MVP はローカル中継アプリで開始するが、後から Cloud Run 構成へ移行しやすいように設計する。

### 16.1 先に固定しておく境界

- Chrome 拡張機能は WebSocket クライアントとして実装する
- 音声送信のメッセージ形式をローカル専用にしすぎない
- 音声認識処理と Google Docs 書き込み処理を同じアプリ内でも責務として分離する

### 16.2 将来変わる可能性が高い部分

- 接続先が `localhost` から Cloud Run の URL に変わる
- 認証方法がローカル前提からクラウド前提に変わる
- ログ保存先がローカルファイルからクラウド基盤へ変わる

### 16.3 変えない前提にしたい部分

- Chrome 拡張機能は音声取得と送信を担当する
- 文字起こしの追記ルールは維持する
- Google ドキュメントを共同 transcript として使う前提は維持する
